# RPH v2.10 S1 四层协议栈修改方案

> **最终版（综合审核意见修订）**：一次性完成四层协议栈代码骨架 + lite/zero/full 协议实现，但 ext 协议保留原 ConformerEngine 不做任何修改。新 Pipeline 体系独立于旧引擎，确保 ext 基线绝对稳定、benchmark 可信。

---

## 1. 设计总纲

### 1.1 四层协议栈的定义

四层协议栈将构象搜索任务按精度和成本划分为四个层次，通过 `ProtocolSpec` 统一管理，由 `ConformerPipeline` 按序执行预定义的 stage 序列。各层使用不同精度的理论方法，执行不同长度的 stage 序列，最终产出符合统一输出契约的结果。

| 协议 | 定位 | 与官方 CENSO 关系 | v2.10 暴露 | 执行引擎 |
|------|------|-------------------|------------|----------|
| **ext** | RPH 内部绝对精度基准：全量 CREST(GFN0→GFN2) + 全量 DFT 优化 + Shermo 热力学 + ωB97X-D4 SP | 超出官方 CENSO | **是** | **原 ConformerEngine** |
| **full** | 官方 CENSO default 近似：Prescreening→Screening→Optimization→Refinement 四阶段漏斗 | 对标官方 CENSO default | **代码就绪，默认不暴露** | 新 ConformerPipeline |
| **lite** | 经济型协议：CREST + r2SCAN-3c 重排（含 mRRHO 热力学校正）+ **无 DFT 几何优化** + 最终 SP | 对标 CENSO-light | **是** | 新 ConformerPipeline |
| **zero** | 极速协议：CREST/xTB + GFN2+mRRHO 排序 + 窄能量窗口选择 + **无 DFT 优化** + 最终 SP | 对标 CENSO-zero | **是** | 新 ConformerPipeline |

**命名规则**：代码中使用 `ext`、`full`、`lite`、`zero`（小写中性名），论文/文档中表述为 *"inspired by the CENSO family"*。

### 1.2 核心原则

1. **ext 使用原 ConformerEngine，完全不动**——ext 是纯净参照系，确保 benchmark 可信
2. **新建 Pipeline 体系**——`ProtocolSpec + ConformerPipeline` 仅服务 lite / zero / full，不侵入 ConformerEngine
3. **一次性交付**——在同一个 PR 中完成 Pipeline 骨架 + lite/zero 协议（可运行）+ full 协议（代码就绪、默认不暴露）+ S2 PES 适配器 + S4 容错
4. **所有协议共享同一输出契约**——对 orchestrator / S2 / S4 透明
5. **每个协议输出附带 `provenance.json`**——语义必须与实际执行路径严格一致，不得虚报
6. **三道验收门强制执行**——ext 等价性门 / 新协议契约门 / 下游稳定性门

### 1.3 为什么 ext 必须保持原样

| 风险场景 | 如果拆分 ext（原方案） | ext 不动（本方案） |
|---|---|---|
| 基线污染 | 拆成 18 个 stage 后，ext 不再是"未动过的旧引擎"，拆分 bug 污染 ext 结果 | ext 调用原 ConformerEngine.run()，零污染 |
| benchmark 失效 | ext 与 lite 的差异无法区分是协议差异还是拆分回归 | ext 是纯净参照系，差异归因可靠 |
| 回归排查 | 需要在 18 个 stage + Pipeline 框架中定位 | 新协议 bug 不影响 ext，排查范围隔离 |
| 交付周期 | 5 天不够稳定（涉及 18 个 stage 抽取 + 旧引擎改造） | 5 天充足（旧引擎零改动，新 stages 独立开发） |

**后续迁移路径**：v2.11+ 可逐步将 ConformerEngine 内部替换为 stage 调用，届时已有 lite/zero/full 作为对照验证。

## 2. FULL 协议完整设计（对标 CENSO 官方标准）

### 2.1 CENSO 四阶段定义

CENSO 的 ensemble optimization 分为 Prescreening, Screening, (Geometry-)Optimization, Refinement 四个步骤，每步使用逐步更精确的设置：

- **Prescreening**：第一轮筛选，用低成本 DFT（PBEh-3c）快速剔除高能构象
- **Screening**：第二轮筛选，用中等精度 DFT（r2SCAN-3c）进行单点重排，引入热力学修正
- **Optimization**：几何优化，仅对 Screening 后保留的少数构象进行完整 DFT 优化
- **Refinement**：最终精修，对优化后的构象进行更高精度单点能计算，并按 Boltzmann 布居截断筛选最终 ensemble

### 2.2 官方默认阈值与方法

| 阶段 | 理论方法 | 基组 | 溶剂模型 | 能量窗口 | 热力学修正 | 其它 |
|------|----------|------|----------|----------|-----------|------|
| Prescreening | PBEh-3c (pbe-d4) | def2-SV(P) | ALPB | **8.0** kcal/mol | — | 宽窗口保召回 |
| Screening | r2SCAN-3c | def2-TZVP | SMD | 3.5 kcal/mol | evaluate_rrho (mRRHO) | — |
| Optimization | r2SCAN-3c | def2-TZVP | SMD | 3.0 kcal/mol | — | 收敛级别 normal |
| Refinement | ωB97X-D4 | def2-TZVPP | SMD | — | — | 最终单点，按 Boltzmann 截断 |

> **审核修正**：Prescreening 能量窗口从 4.0 放宽至 8.0 kcal/mol。原值 4.0 可能过早剔除真实全局最小，官方 CENSO 建议初期保留宽窗口。

## 4. 架构设计：ProtocolSpec + ConformerPipeline

### 4.1 设计动机

四层协议若用四个子类实现，会导致代码膨胀且不灵活。采用 **ProtocolSpec + Pipeline** 模式：

- 新增协议只需定义新 `ProtocolSpec`，无需新增类
- lite / zero / full 共享同一 stage 实现，差异仅在参数和 stage 列表
- ext 走原引擎，与 Pipeline 完全隔离

### 4.2 lite selection_mode 可配置

> **审核修正**：官方 CENSO-light 通常只取 Top-1 进行最终 SP。RPH 的 lite 默认使用 90% 累积布居截断（enhanced-light，针对柔性体系保留更多候选）。用户可按需切换。

```python
# ProtocolSpec.selection_mode 可选值：
#   "cumulative_population" — Boltzmann 累积布居截断（lite 默认，enhanced-light）
#   "top1" — 仅保留能量最低的 1 个构象（官方 CENSO-light 默认）
```

配置：

```yaml
step1:
  protocol_stack:
    lite:
      selection_mode: cumulative_population  # 或 top1
      boltzmann_cutoff: 0.90  # 仅 cumulative_population 时生效
```

### 4.3 ProtocolSpec 数据结构

```python
@dataclass
class ProtocolSpec:
    """协议规格定义——声明式描述一个协议的完整流水线"""
    name: str                          # ext / full / lite / zero
    label: str                         # 人类可读标签
    stages: List[str]                  # 按序执行的 stage 名列表
    params: Dict[str, Any]             # 各阶段参数覆盖
    solvent_map: Dict[str, str]        # stage_name → solvent_model_type (强制)
    provenance: Dict[str, Any]         # 输出语义标记（必须与执行路径严格一致）
    selection_mode: str                # all | top1 | cumulative_population | energy_window
```

`selection_mode` 字段：

| selection_mode | 说明 | 使用协议 |
|---|---|---|
| `all` | 保留全部（ext 默认行为） | ext |
| `cumulative_population` | Boltzmann 累积布居截断 | full, lite (默认) |
| `energy_window` | 窄能量窗口内全部保留 | zero |
| `top1` | 仅保留能量最低的 1 个 | lite (可选) |

### 4.5 ConformerPipeline 执行器

```python
class ConformerPipeline:
    """协议驱动构象搜索流水线——仅服务 lite / zero / full"""

    def __init__(self, spec: ProtocolSpec, config: Dict[str, Any],
                 work_dir: Path, molecule_name: str):
        self.spec = spec
        self.config = config
        self.work_dir = work_dir
        self.molecule_name = molecule_name
        self.state_manager = ConformerStateManager(work_dir / molecule_name, molecule_name)

    def run(self, smiles: str) -> Tuple[Path, float]:
        """执行协议规格定义的全部 stages，返回 (global_min_xyz, energy)"""
        context = PipelineContext(
            smiles=smiles, spec=self.spec, config=self.config,
            work_dir=self.work_dir, molecule_name=self.molecule_name,
            state_manager=self.state_manager,
        )
        for stage_name in self.spec.stages:
            stage = get_stage(stage_name, self.spec)
            context = stage.execute(context)
        return context.global_min_xyz, context.energy
```

## 5. 三道方法学防火墙

### 5.1 lite 重排阶段必须包含热力学修正

**风险**：若 `low_level_rerank` 仅使用 r2SCAN-3c 电子单点能排序，对于柔性前体分子，构象熵的贡献可能使真实全局极小值被排到高能区而被剔除。

**工程要求**：排序依据必须为：

```
G = E_SP(r2SCAN-3c) + G_solv(CPCM) + G_mRRHO(GFN2-xTB)
```

**实现方式**：

对每个候选构象：

1. 运行 r2SCAN-3c SP（ORCA, CPCM 溶剂）
2. 调用 `xtb --hess` 获取单点 Hessian
3. 使用 GFN2-xTB mRRHO 自由能校正（G_mRRHO = G_trans + G_rot + G_vib）
4. 求和得到 G_total

**性能优化**：两步策略——先 r2SCAN-3c SP 粗筛（ΔE < 5 kcal/mol），再仅对窗口内构象做 GFN2 Hessian + mRRHO。

**provenance 降级规则**：若暂未实现 mRRHO，降级为 `"r2scan_3c_sp_only"` / `"none"` 并在文档中明确标注。

### 5.2 zero 协议必须包含 mRRHO（不是纯电子能）

> **审核修正**：zero 的窄窗口选择必须基于 GFN2 mRRHO 自由能，而非纯电子单点能。纯电子能排序的符号翻转率高，即使有窄窗口安全网也不够可靠。

**实现方式**：在 `gfn2_ranking` stage 中，对每个候选构象：

1. 运行 GFN2 SP（`XTBRunner`）
2. 调用 `xtb --hess` 获取 Hessian
3. 计算 mRRHO 自由能校正
4. 排序依据：G = E_GFN2 + G_mRRHO

如果性能不允许全部计算 Hessian，至少对能量窗口内的构象计算。

### 5.3 S2 入口必须实现势能面适配器（强制）

**风险**：lite/zero 输出的几何是 GFN2-xTB 势能面上的平衡结构，而 S2 的 RetroScanner 期望 DFT 级几何。xTB 与 DFT 键长系统性偏差（0.02-0.05 Å）会导致 retro-scan 产生非物理梯度。

**工程设计**（强制实现，非可选）：

```python
# 在 S2 retro_scanner.py 入口处
def adapt_for_pes_mismatch(product_xyz: Path, provenance: Dict) -> Path:
    """
    读取 provenance.json，若几何来自 xTB（has_geometry_optimization=False），
    则对反应中心原子（forming bonds 相关）做轻量级局部 DFT 约束优化。
    """
    if provenance.get("has_geometry_optimization", True):
        return product_xyz  # DFT 几何（ext/full），无需适配

    # 策略 B：对 forming bonds 原子做几圈 DFT 约束优化（推荐）
    adapted = constrained_dft_relax(product_xyz, forming_bonds, max_iter=3)
    return adapted
```

**配置映射**：


## 7. 关键算法细节

### 7.1 玻尔兹曼累积截断（lite / full 专用）

```python
def boltzmann_cumulative_cutoff(
    free_energies: List[float],
    temperature_k: float = 298.15,
    cutoff: float = 0.90,
) -> List[int]:
    """
    按自由能排序，计算 Boltzmann 权重，累加，
    当累积布居 ≥ cutoff 时停止，返回保留的构象索引列表。
    """
    rt = 0.0019872041 * temperature_k
    g_min = min(free_energies)
    weights = [math.exp(-(g - g_min) / rt) for g in free_energies]
    total = sum(weights)
    normalized = [w / total for w in weights]
    cumulative = 0.0
    selected = []
    for idx, w in enumerate(normalized):
        cumulative += w
        selected.append(idx)
        if cumulative >= cutoff:
            break
    return selected
```

### 7.2 窄能量窗口选择（zero 专用）

```python
def energy_window_selection(
    sp_energies: List[float],
    window_kcal: float = 0.5,
) -> List[int]:
    """
    保留 ΔE ≤ window 的所有构象（通常 1-3 个）。
    注意：排序依据是 G_mRRHO（含热力学校正），非纯电子能。
    """
    e_min = min(sp_energies)
    threshold = e_min + window_kcal / 627.509  # kcal/mol → Hartree
    return [idx for idx, e in enumerate(sp_energies) if e <= threshold]
```

### 7.3 lite 协议流水线详解

```text
1. rdkit_embed            → 生成初始 3D 结构
2. crest_sampling         → GFN2 单阶段 CREST 采样
3. isostat_clustering     → 系综聚类去重
4. ensemble_processing    → 筛选候选构象
5. low_level_rerank       → 对每个候选：
   a. r2SCAN-3c SP (CPCM)
   b. GFN2 单点 Hessian → mRRHO 自由能校正
   c. G = E_SP(r2SCAN-3c) + G_solv(CPCM) + G_mRRHO(GFN2-xTB)

   (若 selection_mode=top1 则仅保留最低能量 1 个)

8. write_outputs          → 写出兼容工件 + provenance.json
```

## 14. 三道验收门（强制执行）

### Gate A：ext 等价性门

**触发条件**：任何涉及 ConformerEngine 或 AnchorPhase 的改动。

**验收标准**：

| 检查项 | 标准 | 测试方法 |
|--------|------|----------|
| `product_min.xyz` | 坐标完全一致（RMSD < 1e-6 Å） | diff 坐标文件 |
| 最终 SP 能量 | 差异 < 1e-6 Hartree | 对比输出 |
| `conformer_thermo.csv` | 内容一致 | diff 文件 |
| `conformer_energies.json` | 一致 | diff 文件 |
| `conformer_state.json` | 兼容，旧 checkpoint 可恢复 | 加载测试 |

**不通过**：整个 PR 必须撤回。

### Gate B：新协议输出契约门

**触发条件**：lite / zero 协议实现完成。

**验收标准**（对 lite 和 zero 各选 2 个代表性分子）：

| 检查项 | 标准 |
|--------|------|
| `provenance.json` 存在 | stages_executed 与实际运行一致 |
| `contract_stub` / `has_frequency` 正确 | 语义准确 |
| `conformer_thermo.csv` 可被 S4 读取 | 缺失字段填 NA 不引发异常 |
| `*_global_min.xyz` 格式正确 | S2 可解析 |

### Gate C：下游稳定性门

**触发条件**：Gate A + Gate B 通过后。

**验收标准**（至少运行 1 个完整 S0–S4 pipeline）：

| 检查项 | 标准 |
|--------|------|
| S2 retro-scan | 能生成合理 TS 初猜（PES 适配器生效） |
| S3 TS 优化 | Berny TS 能收敛（或至少不报错） |
| S4 特征提取 | 完成（即使部分特征跳过热力学） |

**不通过**：必须实现/修复 S2 PES 适配器，或调整 xTB 几何策略。

## 16. 关键设计决策总结

| 序号 | 决策 | 优先级 | 理由 |
|------|------|--------|------|
| 1 | ext 使用原 ConformerEngine，完全不动 | **核心决策** | 确保 benchmark 基线绝对稳定 |
| 2 | 新 Pipeline 体系独立于旧引擎 | **核心决策** | 新旧隔离，回归风险可控 |
| 3 | full 代码就绪但默认不暴露 | **核心决策** | 避免未验证方法学风险 |
| 4 | lite selection_mode 可配置（cumulative_population / top1） | **审核修正** | 兼顾官方 CENSO-light 默认行为 |
| 5 | full prescreening 能量窗口 8.0 kcal/mol | **审核修正** | 宽窗口保召回 |
| 6 | zero 必须包含 mRRHO（非纯电子能排序） | **审核修正** | 降低 GFN2 符号翻转风险 |
| 7 | S2 PES 适配器强制实现（默认启用） | **审核修正** | 弥合 xTB/DFT 势能面差异 |
| 8 | 三道验收门强制执行 | **必须** | Gate A/B/C 三道防线 |
| 9 | provenance.json 语义与 stage 路径严格一致 | **必须** | 下游透明性 |
| 10 | 每个 stage 独立文件 + 装饰器注册 | **强烈建议** | 单一职责，便于测试 |
| 11 | 复用 solvent_map.py | **强烈建议** | 不重复造轮子 |

## 17. 一句话结论

> **RPH v2.10 一次性完成四层协议栈代码骨架与 lite/zero 协议实现，ext 协议保留原 ConformerEngine 完全不动以确保 benchmark 基线绝对稳定；新 Pipeline 体系独立于旧引擎，实现 full（代码就绪不暴露）、lite（r2SCAN-3c + mRRHO + 可选 top1）、zero（GFN2 + mRRHO + 窄窗口）三个新协议；三道方法学防火墙（lite/zero/full 强制 mRRHO、S2 强制 PES 适配器）确保化学合理性；三道验收门（ext 等价性 / 新协议契约 / 下游稳定性）确保交付质量。**
